# frank-cli extract generates typed F# from SCXML

> Extracted from [frank-fs/frank#283](https://github.com/frank-fs/frank/issues/283)

**Probability of Successful Implementation: ~82%** (revised up from ~78% — v1 SCXML feature scope now enumerated, resolving the primary ambiguity)

The tagless-final decision simplifies code generation: `frank-cli` emits polymorphic functions `TransitionAlgebra<'r> -> 'r` that call algebra fields in sequence. The SCXML-to-DU-types generation (AC-1) is mechanical. AC-4 (no Frank runtime dependencies beyond Core) is clean now that algebra types live in `Frank.Statecharts.Core` (no separate Abstractions package). The remaining risk is AC-2 (semantic equivalence to model.bin) for AND-state and history transitions.

## Thesis

`frank-cli extract` should emit a single typed F# source file — discriminated unions for events, states, regions, and roles, plus transition programs as polymorphic functions taking `TransitionAlgebra<'r>` — as a build artifact generated into `obj/`. This replaces the current `model.bin` binary blob with compile-time-checked code that the compiler, IDE, and analyzers can all reason about.

## Problem

Today, `frank-cli` extracts statechart metadata into `model.bin`, a binary artifact loaded at runtime. This means:

* Event names are strings — typos compile fine and fail at runtime
* State/region references are unchecked — no exhaustiveness guarantees
* Analyzers cannot validate statechart usage without taking a dependency on the statechart runtime
* IDE tooling (IntelliSense, refactoring) is blind to the statechart vocabulary
* The developer gets no compile-time feedback when the SCXML changes

## V1 SCXML Feature Scope

| Feature | V1 | Notes |
|---------|-----|-------|
| `<state>`, `<parallel>`, `<final>` | Yes | Map to XOR, AND composites and final states |
| `<transition>` with `event`, `target`, `cond` | Yes | Becomes algebra calls (Exit, Enter, Fork) |
| `<history>` (shallow and deep) | Yes | `RestoreHistory` algebra call |
| `<initial>` / `initial` attribute | Yes | Default entry for composites |
| `<datamodel>`, `<data>` | No — v2 | Context type is user-defined in F#, not generated from SCXML |
| `<script>`, `<assign>` | No — v2 | Executable content requires an expression evaluator |
| `<invoke>`, `<send>` | No — v2 | Service invocation is out of scope |
| `<onentry>`, `<onexit>` | No — v2 | Entry/exit actions are handler concerns, not algebra concerns |
| `<donedata>`, `<content>` | No — v2 | Data payloads are user-defined |
| `<if>`, `<elseif>`, `<else>` | No — v2 | Conditional logic belongs in F# handlers |

Note: `<onentry>`/`<onexit>` are SCXML **side-effect actions** triggered when entering/exiting a state. These are distinct from the algebra's `Exit`/`Enter` operations, which change state configuration. The algebra operations are fully in scope; the SCXML action elements are v2.

## Solution

Add `--format fsharp` to `frank-cli extract`. The CLI reads SCXML and emits a single `<Name>.Generated.fs` file into the project's intermediate output directory. This file is compiled into the assembly automatically via MSBuild targets (#284). F# top-to-bottom ordering within the file handles dependencies naturally (types precede programs). Users who want to split or customize can pull the generated code into their own files.

**Naming conflicts:** If two SCXML files produce the same PascalCased module name, `frank extract` errors with a clear message. The caller refines the inputs (SCXML, ALPS, etc.) to resolve the conflict.

**Module naming convention:** For an SCXML file named `OrderStatechart.scxml`, the generated module is `OrderStatechart.Generated` (filename sans extension, PascalCased, with `.Generated` suffix). The `.Generated` suffix is the convention that `Frank.Analyzers` (#285) uses to identify generated types (FRANK105 detection contract). DU types within use `[<RequireQualifiedAccess>]`.

The generated output includes:

1. **DU types** for events, states, regions, and roles — enabling exhaustive pattern matching and compiler-checked references
2. **Transition programs** as polymorphic functions `TransitionAlgebra<'r> -> 'r` — the mechanical translation of SCXML transitions into typed algebra calls. LCA is computed at generation time and baked into the program as a parameter.
3. **Hierarchy metadata** expressing parent/child region relationships as data — consumable by the middleware and analyzers without runtime reflection

Example generated output:

```fsharp
// Auto-generated by frank-cli — do not edit
[<AutoOpen>]
module OrderStatechart.Generated

open Frank.Statecharts.Ast  // TransitionAlgebra<'r>, HistoryKind, ActiveStateConfiguration

[<RequireQualifiedAccess>]
type Event =
    | PickCompleted
    | PackCompleted
    | ShipCompleted

[<RequireQualifiedAccess>]
type State =
    | Created
    | Fulfilling
    | Shipped

[<RequireQualifiedAccess>]
type Region =
    | Main
    | Fulfillment

/// Transition: Picking → Packing (triggered by PickCompleted)
let onPickCompleted<'r> (lca: string) (alg: TransitionAlgebra<'r>) : 'r =
    alg.Exit "Picking"
    |> alg.Sequence (alg.RecordHistory "Fulfillment")
    |> alg.Sequence (alg.Enter "Packing")
```

The generated code is never checked into source control. It lives in `obj/` and is regenerated on every build when the SCXML changes.

## Architectural constraints

* Generated `.fs` file MUST appear before all user-authored `.fs` files in compilation order (F# single-pass compiler constraint)
* Generated transition programs MUST be polymorphic functions `TransitionAlgebra<'r> -> 'r` — no direct `ActiveStateConfiguration` manipulation
* Generated DU types MUST use `[<RequireQualifiedAccess>]` to avoid name collisions with user code
* Generated code MUST NOT depend on any Frank types beyond `Frank.Statecharts.Core` (`TransitionAlgebra<'r>`, `HistoryKind`, `ActiveStateConfiguration`)
* `model.bin` MAY continue to be emitted alongside for backward compatibility; the `.fs` output is the primary artifact going forward

## Acceptance Criteria

### AC-1: Generated DU types enable exhaustive pattern matching

```
Given: an SCXML with events PickCompleted, PackCompleted, ShipCompleted
When: frank-cli extract --format fsharp generates the Event DU
Then: a match expression on Event that omits ShipCompleted produces a
  compiler warning (incomplete pattern match)
Falsifiable by: adding an event to the SCXML, regenerating, and verifying
  that all existing match expressions produce warnings until updated
```

### AC-2: Generated transition programs are semantically equivalent to model.bin interpretation

```
Given: an SCXML statechart with XOR, AND, and history transitions
When: the same statechart is loaded via model.bin (current path)
  AND via generated transition programs (new path)
Then: both paths produce identical ActiveStateConfiguration results for the
  same input event sequences
Falsifiable by: any divergence between the two paths means the code generator
  does not faithfully represent the SCXML semantics
```

### AC-3: Generated hierarchy metadata is consumable without runtime reflection

```
Given: an SCXML with parent/child region relationships (Order → Pick, Pack, Ship)
When: frank-cli extract generates hierarchy metadata
Then: the middleware can resolve parent/child relationships at startup by
  reading the generated types — no Assembly.GetTypes() or attribute scanning
Falsifiable by: the middleware requiring reflection to discover the hierarchy
```

### AC-4: Generated code compiles with only Frank.Statecharts.Core

```
Given: a generated .fs file from frank-cli extract
When: the file is compiled in a project that references only
  Frank.Statecharts.Core
Then: compilation succeeds
Falsifiable by: the generated code requiring a reference to Frank.Statecharts
  or any other Frank package
```

### AC-5: Regeneration after SCXML change produces compile errors at stale usage sites

```
Given: a project using generated Event.PickCompleted in handler code
When: the SCXML is modified to rename PickCompleted to PickFinished
  AND frank-cli extract regenerates the .fs file
Then: the handler code fails to compile with a name resolution error
  referencing PickCompleted
Falsifiable by: the handler compiling successfully after the rename
```

### AC-6: Generated transition programs are polymorphic

```
Given: a generated transition program (e.g., onPickCompleted)
When: the program is called with a test stub algebra
Then: the stub algebra's operations are invoked in the correct sequence
  AND the program's type is TransitionAlgebra<'r> -> 'r
Falsifiable by: the generated program using any mechanism other than
  TransitionAlgebra field calls
```

### AC-7: Unsupported SCXML features produce warnings, not failures

```
Given: an SCXML file containing <invoke> and <script> elements
When: frank-cli extract --format fsharp processes the file
Then: the generator emits a warning per unsupported element (to stderr)
  AND generates code for all supported elements
  AND the generated code compiles successfully
Falsifiable by: the generator failing with an error or silently ignoring
  unsupported elements without warning
```
