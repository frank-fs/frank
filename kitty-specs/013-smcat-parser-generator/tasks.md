# Work Packages: smcat Parser and Generator

**Inputs**: Design documents from `/kitty-specs/013-smcat-parser-generator/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/api-signatures.md, quickstart.md

**Tests**: Tests are explicitly required by the spec (SC-009 golden file tests, Expecto test files mirroring WSD pattern).

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package is independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `/tasks/` generated by `/spec-kitty.tasks`. Treat this file as the high-level checklist; keep deep implementation detail inside the prompt files.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- Include precise file paths or modules.

## Path Conventions
- **Source**: `src/Frank.Statecharts/Smcat/`

## Work Package WP02: Lexer (Priority: P0)

**Goal**: Implement the smcat tokenizer that converts raw smcat text into a flat token stream, plus lexer tests.
**Independent Test**: `dotnet test --filter "Smcat.Lexer"` passes for all token types including edge cases (comments, quoted strings, attributes, activities, line endings).
**Prompt**: `/tasks/WP02-lexer.md`
**Estimated Prompt Size**: ~450 lines

### Included Subtasks
- [ ] T006 Create `src/Frank.Statecharts/Smcat/Lexer.fs` with mutable scanning pattern (matching WSD Lexer)
- [ ] T007 Implement identifier and quoted string tokenization (plain identifiers, dot-separated identifiers, quoted `"strings"`)
- [ ] T008 Implement punctuation and arrow tokenization (=>, :, ;, ,, [, ], {, }, /, =)
- [ ] T009 Implement comment handling (# lines), whitespace skipping, and newline handling (\r\n and \n)
- [ ] T010 Implement activity keyword detection (entry/, exit/, ...) and caret/close-bracket prefix tokens
- [ ] T011 [P] Create `test/Frank.Statecharts.Tests/Smcat/LexerTests.fs` with comprehensive token tests

### Implementation Notes
- Follow WSD Lexer.fs mutable scanning pattern: `pos`, `line`, `col` variables, `peek()`, `advance()`, `newline()`, `makeToken()` inline helpers
- Identifier chars: alphanumeric, underscore, dot (for deep.history); hyphens allowed in middle of identifiers
- Quoted strings: text between double quotes, preserve content without quotes
- `entry/` and `exit/` must be detected as single tokens (not Identifier + ForwardSlash)
- `...` (ellipsis) is the do-activity marker
- `=>` is the transition arrow (not `=` followed by `>`)

### Parallel Opportunities
- T011 (tests) can be written in parallel with T006-T010 if the Token type contract is stable.

### Dependencies
- Depends on WP01 (Types.fs must exist for Token/TokenKind types).

### Risks & Mitigations
- Context-dependent `]` token (close bracket vs fork/join prefix): Handle by emitting `CloseBracketPrefix` only at statement start position (column 1 or after newline/semicolon). Mid-expression `]` is always `RightBracket`.
- `entry/` vs identifier `entry` followed by `/`: Use lookahead -- if identifier is "entry" or "exit" and next char is `/`, emit the combined activity token.

## Work Package WP03: LabelParser & Parser (Priority: P1)

**Goal**: Implement transition label parsing and the full recursive-descent parser that produces SmcatDocument from a token stream.
**Independent Test**: `dotnet test --filter "Smcat.LabelParser or Smcat.Parser"` passes; the onboarding example from quickstart.md parses correctly.
**Prompt**: `/tasks/WP03-label-parser-and-parser.md`
**Estimated Prompt Size**: ~500 lines

### Included Subtasks
- [ ] T012 Create `src/Frank.Statecharts/Smcat/LabelParser.fs` implementing `parseLabel` function for `event [guard] / action` format
- [ ] T013 Create `src/Frank.Statecharts/Smcat/Parser.fs` with ParserState type and core parsing infrastructure (token consumption, error recovery, lookahead)
- [ ] T014 Implement state declaration parsing (explicit state lists with commas, state activities, state attributes)
- [ ] T015 Implement transition parsing (source => target with optional label and attributes)
- [ ] T016 Implement composite state parsing (recursive { ... } blocks with nested SmcatDocument)
- [ ] T017 Implement `parseSmcat` convenience function (tokenize + parse in one call)
- [ ] T018 [P] Create `test/Frank.Statecharts.Tests/Smcat/LabelParserTests.fs` with edge case tests for all label component combinations
- [ ] T019 [P] Create `test/Frank.Statecharts.Tests/Smcat/ParserTests.fs` with full parse tests (states, transitions, composites, pseudo-states)

### Implementation Notes
- LabelParser grammar: `label ::= event? guard? action?` where event = text before `[` or `/`, guard = text between `[` and `]`, action = text after `/`
- Parser follows WSD recursive-descent pattern: ParserState with mutable position, elements, errors, warnings
- Attribute parsing: `[key=value key2="quoted value"]` -- reuse bracket tokens, parse key=value pairs inside
- Composite states: when `{` encountered after state identifier, recursively parse inner document until matching `}`
- Statement terminators: both `;` and `,` end a statement
- Empty input, whitespace-only, comment-only inputs must produce valid empty SmcatDocument
- Pseudo-state inference: call `inferStateType` from Types.fs when constructing SmcatState nodes

### Parallel Opportunities
- T018 (LabelParser tests) and T019 (Parser tests) can be written in parallel with implementation if contracts are stable.
- T012 (LabelParser) and T013 (Parser infrastructure) can proceed in parallel since LabelParser is self-contained.

### Dependencies
- Depends on WP02 (Lexer must exist to produce tokens).

### Risks & Mitigations
- Disambiguation between state declaration and transition: When parser sees an identifier, it must lookahead for `=>` (transition) vs `,`/`;`/`{`/`[`/activity keywords (state declaration).
- Deeply nested composite states (5+ levels): The recursive approach handles this naturally; add depth counter with warning at 50+ levels.

## Work Package WP04: Structured Error Reporting (Priority: P2)

**Goal**: Implement comprehensive error recovery and structured failure reports for malformed smcat input (User Story 5).
**Independent Test**: `dotnet test --filter "Smcat.Error"` passes; malformed inputs produce ParseFailure records with position, expected/found, and corrective examples.
**Prompt**: `/tasks/WP04-error-reporting.md`
**Estimated Prompt Size**: ~350 lines

### Included Subtasks
- [ ] T020 Enhance Parser.fs with error recovery: skip-to-next-statement on parse failure, continue collecting elements and errors
- [ ] T021 Implement corrective example generation for common error patterns (missing colon, invalid arrow, unclosed brackets, unclosed composite blocks)
- [ ] T022 Implement configurable error limit (maxErrors parameter, default 50) with ErrorLimitReached flag
- [ ] T023 Add ParseWarning generation for ambiguous constructs (e.g., state name that looks like pseudo-state but has explicit type attribute)
- [ ] T024 [P] Create `test/Frank.Statecharts.Tests/Smcat/ErrorTests.fs` with tests for all acceptance scenarios from User Story 5

### Implementation Notes
- Error recovery strategy: on parse failure, record ParseFailure, then advance to the next `;`, `,`, or `Newline` token and resume parsing
- Corrective examples should show the correct syntax for the detected error pattern (e.g., missing colon -> `source => target: event;`)
- Multiple errors collected up to maxErrors (default 50 per FR-011)
- ParseResult.Document should still contain successfully parsed elements (best-effort partial result)

### Parallel Opportunities
- T024 (ErrorTests) can be written in parallel with T020-T023.

### Dependencies
- Depends on WP03 (Parser must exist with basic parsing before error recovery is layered in).

### Risks & Mitigations
- Error recovery can cause cascading false positives: mitigate by skipping to a clean synchronization point (statement terminator) before resuming.
- Corrective examples must stay up-to-date with grammar changes: keep examples as string constants close to the error detection logic.

## Work Package WP05: Generator (Priority: P2)

**Goal**: Implement smcat text generation from StateMachineMetadata (User Story 3), plus generator tests.
**Independent Test**: `dotnet test --filter "Smcat.Generator"` passes; generated smcat text for known metadata is valid and matches expected output.
**Prompt**: `/tasks/WP05-generator.md`
**Estimated Prompt Size**: ~400 lines

### Included Subtasks
- [ ] T025 Create `src/Frank.Statecharts/Smcat/Generator.fs` implementing the `generate` function with the API from contracts/api-signatures.md
- [ ] T026 Implement initial state handling: emit `initial => <first_state>;` as first transition when initial state exists
- [ ] T027 Implement transition label formatting: `event [guard] / action` with optional components omitted cleanly
- [ ] T028 Implement final state handling: emit `<source> => final;` transitions for states with IsFinal flag
- [ ] T029 [P] Create `test/Frank.Statecharts.Tests/Smcat/GeneratorTests.fs` with tests covering all User Story 3 acceptance scenarios

### Implementation Notes
- Generator API takes `StateMachineMetadata<'State, 'Event, 'Context>` plus helper functions (`stateNames`, `eventNames`) and an explicit `transitions` list (since metadata stores transitions as closures)
- Output format: one statement per line, semicolon terminators, no trailing commas
- Ordering: initial transition first, regular transitions in provided order, final transitions last
- Label formatting rules: event only -> `event`, event + guard -> `event [guard]`, event + guard + action -> `event [guard] / action`, guard only -> `[guard]`, action only -> `/ action`
- Module declaration: `module internal Frank.Statecharts.Smcat.Generator`
- Needs access to `StateMachineMetadata` from `Frank.Statecharts.Types` (already in compile scope)

### Parallel Opportunities
- T029 (GeneratorTests) can be written in parallel with T025-T028.
- WP05 is independent of WP04 (error reporting) -- they can run in parallel after WP03 completes.

### Dependencies
- Depends on WP01 (Types.fs for TransitionLabel, SmcatAttribute types).
- Does NOT depend on WP02/WP03/WP04 (generator works from StateMachineMetadata, not from parsed AST).

### Risks & Mitigations
- StateMachineMetadata generic type parameters require the caller to provide name-conversion functions -- document this clearly in the API.
- Generated output must be valid smcat that re-parses correctly (validated by roundtrip tests in WP06).

## Work Package WP06: Roundtrip Tests & Golden Files (Priority: P2)

**Goal**: Implement roundtrip consistency tests (User Story 4) and golden file tests (SC-009) that validate parse-generate-reparse equivalence.
**Independent Test**: `dotnet test --filter "Smcat.RoundTrip"` passes; at least 3 golden file examples of varying complexity round-trip successfully.
**Prompt**: `/tasks/WP06-roundtrip-tests-and-golden-files.md`
**Estimated Prompt Size**: ~400 lines

### Included Subtasks
- [ ] T030 Create golden file smcat examples: simple linear (3 states, 3 transitions), branching with guards, composite states
- [ ] T031 Create `test/Frank.Statecharts.Tests/Smcat/RoundTripTests.fs` with parse-map-generate-reparse cycle tests
- [ ] T032 Implement semantic equivalence comparison for SmcatDocument ASTs (state names, transition topology, labels match; ordering may differ)
- [ ] T033 Add roundtrip tests for edge cases: guards-only transitions, actions-only transitions, pseudo-states, empty input
- [ ] T034 Validate SC-005 (roundtrip consistency) and SC-009 (golden file tests) success criteria are met

### Implementation Notes
- Golden files can be embedded as string literals in the test file (following WSD pattern) or as .smcat files in a test data directory
- Semantic equivalence: compare sets of (stateName, stateType) pairs and sets of (source, target, event, guard, action) transition tuples
- Roundtrip cycle: parse smcat -> use generator with extracted state/event/transition info -> re-parse generated text -> compare ASTs
- At least 3 golden files required by SC-009: (1) simple linear flow, (2) branching with guards/actions, (3) composite/nested states

### Parallel Opportunities
- T030 (golden files) can be written immediately since it's just smcat text content.

### Dependencies
- Depends on WP03 (Parser) and WP05 (Generator) -- both must be complete for roundtrip cycle.

### Risks & Mitigations
- Ordering differences between original and roundtripped AST: Compare by topology (sets) not by position.
- Generator may not reproduce all smcat features (e.g., state activities, attributes) since StateMachineMetadata doesn't carry them: document which features survive roundtrip.

## Dependency & Execution Summary

- **Sequence**: WP01 (Types) -> WP02 (Lexer) -> WP03 (Parser) -> WP04 (Error Reporting)
- **Parallel after WP01**: WP05 (Generator) can start after WP01 since it depends on Types.fs and StateMachineMetadata, not on the parser.
- **Parallel after WP03**: WP04 (Error Reporting) and WP05 (Generator) can run in parallel.
- **Final**: WP06 (Roundtrip) requires both WP03 (Parser) and WP05 (Generator).
- **Not included**: Mapper (Smcat/Mapper.fs) is blocked on spec 020 and excluded from this task breakdown.

```
WP01 (Types) ──> WP02 (Lexer) ──> WP03 (Parser) ──> WP04 (Errors)
     │                                    │                │
     └──> WP05 (Generator) ───────────────┴──> WP06 (Roundtrip)
```

- **MVP Scope**: WP01 + WP02 + WP03 deliver the core parser (User Story 1). Adding WP05 delivers generation (User Story 3). WP06 validates the roundtrip guarantee (User Story 4).
- **Parallelization**: WP05 can start as soon as WP01 completes. After WP03 completes, WP04 and WP06 (if WP05 is done) can proceed in parallel.
