## Work Package WP03: StatefulResource Computation Expression (Priority: P0)

**Goal**: Implement `statefulResource` CE with `inState` blocks that register state-specific HTTP method handlers.
**Independent Test**: CE compiles and produces a `Resource` with `StateMachineMetadata` in endpoint metadata.
**Prompt**: `tasks/WP03-stateful-resource-ce.md`
**Estimated Size**: ~500 lines

### Included Subtasks
- [x] T010 Create `src/Frank.Statecharts/StatefulResourceBuilder.fs` with `StatefulResourceSpec` and `StatefulResourceBuilder` CE
- [x] T011 Implement `machine` custom operation for defining `StateMachine<'S,'E,'C>` inline
- [x] T012 Implement `inState` custom operation for per-state handler registration
- [x] T013 Implement `onTransition` custom operation for observable transition hooks
- [x] T014 Implement `resolveInstanceId` custom operation for configuring instance key extraction from route parameters
- [x] T015 Implement `Build` method that produces `Resource` with `StateMachineMetadata`

### Implementation Notes
- The CE wraps `ResourceBuilder` rather than extending it (see plan DD-01)
- `StatefulResourceSpec` accumulates: machine definition, per-state handler map, transition observers, instance ID resolver
- At build time: all handlers from all `inState` blocks are registered as endpoints, plus `StateMachineMetadata` is added to each endpoint's metadata
- The `StateMachineMetadata` contains: boxed `StateMachine`, per-state allowed methods map, store factory
- Follow `ResourceBuilder`'s `Yield`/`Run` pattern for the CE
- `inState` blocks need to collect HTTP method handlers scoped to a specific state value

### Parallel Opportunities
- None within this WP (all subtasks are sequential -- CE must build incrementally)

### Dependencies
- Depends on WP01 (core types)

### Risks & Mitigations
- F# CE limitations: nested CEs (`machine { ... }` inside `statefulResource { ... }`) may require separate builder types. Prototype early; fall back to method-chain API if CE nesting is too complex.
- Type inference: `inState` blocks need to know the state type. May need explicit type annotation on first use.

## Work Package WP04: State-Aware Middleware (Priority: P0)

**Goal**: Implement middleware that intercepts requests to stateful resources, evaluates state/guards, and routes to the correct state-specific handler or returns appropriate HTTP status codes.
**Independent Test**: Middleware correctly returns 405 for disallowed methods, evaluates guards, and returns correct status codes for blocked transitions.
**Prompt**: `tasks/WP04-state-aware-middleware.md`
**Estimated Size**: ~450 lines

### Included Subtasks
- [x] T016 Create `src/Frank.Statecharts/Middleware.fs` with state-aware request interception
- [x] T017 Implement state lookup: retrieve current state from `IStateMachineStore` using resolved instance ID
- [x] T018 Implement method filtering: check if HTTP method is allowed in current state, return 405 if not
- [x] T019 Implement guard evaluation: evaluate guards in order, map `BlockReason` to HTTP status codes
- [x] T020 Implement transition execution: on successful handler completion, apply transition and fire `onTransition` hook
- [x] T021 Create `test/Frank.Statecharts.Tests/MiddlewareTests.fs` with integration tests

### Implementation Notes
- Follow `Frank.LinkedData`'s middleware pattern: check for `StateMachineMetadata` marker -> intercept or pass through
- Guard evaluation order: registered order, first `Blocked` short-circuits
- BlockReason -> HTTP status code mapping (defined in Types.fs):
  - `NotAllowed` -> 403, `NotYourTurn` -> 409, `InvalidTransition` -> 400, `PreconditionFailed` -> 412, `Custom(code, msg)` -> code
- State lookup must handle missing instance (new resource): use `StateMachine.Initial` as default
- `onTransition` hook fires after successful state update, not before
- Middleware must resolve `IStateMachineStore` from DI

### Parallel Opportunities
- T021 (tests) can be scaffolded once T016 interface is defined

### Dependencies
- Depends on WP01 (types), WP02 (store), WP03 (CE produces metadata)

### Risks & Mitigations
- Middleware ordering: must run after routing but before endpoint execution. Document in WebHostBuilderExtensions.
- Concurrent requests to same instance: MailboxProcessor serializes, but middleware must await store operations

**Goal**: Create `[<AutoOpen>]` extension modules for `WebHostBuilder` and `ResourceBuilder` following Frank's convention pattern.
**Independent Test**: `useStatecharts` registers middleware and DI services; extensions compose with existing Frank extensions (Auth, LinkedData).
**Prompt**: `tasks/WP05-extension-modules.md`
**Estimated Size**: ~350 lines

### Included Subtasks
- [x] T022 Create `src/Frank.Statecharts/ResourceBuilderExtensions.fs` with `stateMachine` custom operation on `ResourceBuilder`
- [x] T023 Create `src/Frank.Statecharts/WebHostBuilderExtensions.fs` with `useStatecharts` custom operation
- [x] T024 Implement DI registration in `useStatecharts`: register `IStateMachineStore` (default `MailboxProcessorStore`) and middleware
- [x] T025 Verify compilation order in .fsproj (Types -> Store -> StatefulResourceBuilder -> Middleware -> Extensions)

### Implementation Notes
- `ResourceBuilderExtensions`: adds `stateMachine` operation to standard `ResourceBuilder` for cases where full `statefulResource` CE is not needed (simple metadata-only annotation)
- `WebHostBuilderExtensions`: adds `useStatecharts` that registers:
  1. `IStateMachineStore<'S,'C>` in DI (default: `MailboxProcessorStore`)
  2. Middleware via `plug` (runs after routing, before endpoint execution — same as LinkedData)
- Follow `Frank.Auth.WebHostBuilderExtensions` and `Frank.LinkedData.WebHostBuilderExtensions` exactly
- `useStatecharts` should accept optional configuration callback for custom store registration

### Parallel Opportunities
- T022 and T023 can proceed in parallel (different files, different extension targets)

### Dependencies
- Depends on WP03 (CE), WP04 (middleware)

### Risks & Mitigations
- Middleware ordering with LinkedData/Auth: document recommended order (Auth -> Statecharts -> LinkedData)

## Work Package WP06: Integration Tests & Tic-Tac-Toe Validation (Priority: P1)

**Goal**: End-to-end integration tests using TestHost, including a simplified tic-tac-toe state machine to validate the full stack.
**Independent Test**: Complete request lifecycle works: state-dependent methods, guard evaluation, transition hooks, filtered affordances.
**Prompt**: `tasks/WP06-integration-tests.md`
**Estimated Size**: ~450 lines

### Included Subtasks
- [ ] T026 Create `test/Frank.Statecharts.Tests/StatefulResourceTests.fs` with TestHost-based integration tests
- [ ] T027 Implement simplified tic-tac-toe state machine as test fixture (XTurn/OTurn/Won/Draw states)
- [ ] T028 Test state-dependent method availability (POST allowed in XTurn, not in Won)
- [ ] T029 Test guard evaluation with mock ClaimsPrincipal (turn-based, role-based guards)
- [ ] T030 Test transition hooks fire correctly (observable receives old/new state)
- [ ] T031 Test filtered affordances (response includes only available transitions per state)

### Implementation Notes
- Use ASP.NET Core TestHost (`Microsoft.AspNetCore.TestHost`) like existing Frank tests
- Simplified tic-tac-toe: 4 states (XTurn, OTurn, Won, Draw), 1 event (MakeMove), 2 guards (isPlayerX, isPlayerO)
- Test the full middleware pipeline: request -> state lookup -> guard -> handler -> transition -> hook
- Verify HTTP status codes: 200 (success), 405 (wrong method for state), 409 (guard blocked), 400 (invalid transition)
- Test store lifecycle: create instance, transition through states, verify final state

### Parallel Opportunities
- T027 (test fixture) can be written first; T028-T031 depend on it but are independent of each other

### Dependencies
- Depends on WP04 (middleware), WP05 (extensions)

### Risks & Mitigations
- TestHost setup complexity: reference existing Frank test projects for patterns
- ClaimsPrincipal mocking: use standard ASP.NET Core test authentication
