**Independent Test**: Given various F# record and DU types, `deriveShape` produces correct `ShaclShape` with appropriate property shapes, datatypes, cardinality, and nested references.
**Prompt**: `tasks/WP02-type-mapping-and-shape-derivation.md`
**Estimated Size**: ~500 lines
**Requirement Refs**: FR-001, FR-002, FR-003, FR-004, FR-005, FR-006, FR-007, FR-017, FR-020

### Included Subtasks
- [x] T006 Create `src/Frank.Validation/TypeMapping.fs` with `mapType : Type -> XsdDatatype option` and `xsdUri : XsdDatatype -> Uri`
- [x] T007 Implement record field derivation: `PropertyInfo -> PropertyShape` with datatype, minCount/maxCount
- [x] T008 Implement option type unwrapping (`option<T>` -> minCount 0, inner type mapping)
- [x] T009 Implement DU derivation: simple cases -> `sh:in`, payload cases -> `sh:or` with per-case NodeShapes
- [x] T010 Implement nested record handling: `sh:node` references to child NodeShapes
- [x] T011 Implement recursive type cycle detection with configurable depth limit (default 5)
- [x] T012 Implement generic type expansion at point of use (e.g., `PagedResult<Customer>` -> concrete shape)
- [x] T013 Create `test/Frank.Validation.Tests/TypeMappingTests.fs` with mapping tests for all supported F# types
- [x] T014 Create `test/Frank.Validation.Tests/ShapeDerivationTests.fs` with derivation tests for records, DUs, nested/recursive types

### Implementation Notes
- `TypeMapping.mapType` is a pure function: `Type -> XsdDatatype option` (None for non-primitive types that need sh:node)

- `isDerivableType` excludes HttpContext, HttpRequest, CancellationToken, etc.
- Use `FSharp.Reflection.FSharpType` for IsRecord/GetRecordFields, IsUnion/GetUnionCases
- Collection types (`list<T>`, `T[]`, `seq<T>`) produce list-level cardinality constraints
- Guid auto-derives `sh:pattern` for UUID format

## Work Package WP03: Validator & Validation Middleware (Priority: P0)

**Goal**: Implement `Validator.fs` (dotNetRdf SHACL validation execution) and `ValidationMiddleware.fs` (ASP.NET Core middleware for request interception). Middleware intercepts after auth, validates against derived shapes, short-circuits with 422 on failure.
**Independent Test**: Invalid requests return 422 before handler executes; valid requests pass through to handler. Middleware correctly reads `ValidationMarker` from endpoint metadata.
**Prompt**: `tasks/WP03-validator-and-middleware.md`
**Estimated Size**: ~500 lines
**Requirement Refs**: FR-008, FR-009

### Included Subtasks
- [x] T015 Create `src/Frank.Validation/Validator.fs` with `validate : ShaclShape -> IGraph -> ValidationReport`
- [x] T016 Implement ShaclShape -> dotNetRdf `ShapesGraph` conversion (shapes graph construction)
- [x] T017 Implement request data -> dotNetRdf `IGraph` conversion (data graph construction from JSON body / query params)
- [x] T018 Create `src/Frank.Validation/ValidationMiddleware.fs` with endpoint metadata check, deserialization, validation, and short-circuit
- [x] T019 Implement query parameter validation for GET requests
- [x] T020 Create `test/Frank.Validation.Tests/ValidatorTests.fs` with pass/fail validation tests using dotNetRdf
- [x] T021 Create `test/Frank.Validation.Tests/MiddlewareTests.fs` with TestHost-based pipeline tests

### Implementation Notes
- Use `VDS.RDF.Shacl.ShapesGraph` for validation: construct once at startup, reuse per request
- Data graph construction: deserialize JSON body to `IGraph` using property paths from the shape
- Middleware checks `endpoint.Metadata.GetMetadata<ValidationMarker>()` -- if null, pass through (zero overhead)
- Pipeline ordering: runs after `useAuth`, before handler dispatch
- Missing body on validated endpoint -> validation failure (not deserialization error)
- Handler framework types (HttpContext, etc.) -> skip validation with startup warning if `validate` was explicitly set

### Parallel Opportunities
- T020 (ValidatorTests) can be scaffolded once T015/T016 are defined
- T021 (MiddlewareTests) depends on T018 but can parallelize with T019

### Dependencies
- Depends on WP01 (types), WP02 (shape derivation, type mapping)

### Risks & Mitigations
- dotNetRdf SHACL validation performance: cache ShapesGraph, benchmark with 20-property shapes
- Data graph construction from JSON: map JSON properties to RDF predicates using shape's property paths

**Prompt**: `tasks/WP04-violation-reporting.md`

## Work Package WP05: Capability-Dependent Shape Resolution (Priority: P2)

**Goal**: Implement `ShapeResolver.fs` for runtime shape selection based on Frank.Auth `ClaimsPrincipal` capabilities. Admins get permissive shapes; regular users get restrictive shapes. Same request body, different validation outcomes based on principal.
**Independent Test**: Admin principal passes validation with unrestricted values; regular user principal fails validation for the same values. Base shape applied when no override matches.
**Prompt**: `tasks/WP05-capability-dependent-shapes.md`
**Estimated Size**: ~350 lines
**Requirement Refs**: FR-013, FR-014

### Included Subtasks
- [x] T027 Create `src/Frank.Validation/ShapeResolver.fs` with `resolve : ShapeResolverConfig -> ClaimsPrincipal -> ShaclShape`
- [ ] T028 Implement claim-based override matching (iterate overrides, first match wins)
- [ ] T029 Implement base shape fallback when no override matches
- [ ] T030 Integrate ShapeResolver into ValidationMiddleware (read principal from HttpContext.User)
- [ ] T031 Create `test/Frank.Validation.Tests/CapabilityTests.fs` with capability-dependent validation tests

### Implementation Notes
- ShapeResolver is a pure function: `ShapeResolverConfig -> ClaimsPrincipal -> ShaclShape`
- Override matching: iterate `ShapeResolverConfig.Overrides` in registration order, check `RequiredClaim` tuple against principal's claims
- Base shape (most restrictive) is the auto-derived shape without overrides
- Middleware integration: if `ValidationMarker.ResolverConfig` is `Some`, use `ShapeResolver.resolve`; otherwise use the base shape
- `ClaimsPrincipal` comes from `HttpContext.User` (populated by Frank.Auth middleware upstream)

### Parallel Opportunities
- T027/T028/T029 (resolver logic) can proceed in parallel with T030 (middleware integration)
- T031 (tests) requires both

### Dependencies
- Depends on WP01 (types), WP02 (shape derivation), WP03 (middleware)

### Risks & Mitigations
- Frank.Auth integration: read-only dependency on `ClaimsPrincipal` -- no coupling to `AuthRequirement` internals
- Override ordering: document that first-match-wins; recommend most-specific overrides first

## Work Package WP06: Custom Constraints & Conflict Detection (Priority: P2)

**Goal**: Implement `ShapeMerger.fs` for merging developer-supplied custom SHACL constraints with auto-derived shapes. Custom constraints are additive only. Detect contradictions at startup and raise `InvalidOperationException`.
**Independent Test**: Custom sh:pattern on a string field is enforced alongside auto-derived constraints. Attempting to weaken a required field raises startup error. Cross-field SPARQL constraints validate correctly.
**Prompt**: `tasks/WP06-custom-constraints.md`
**Estimated Size**: ~400 lines
**Requirement Refs**: FR-015, FR-016

### Included Subtasks
- [x] T032 Create `src/Frank.Validation/ShapeMerger.fs` with `merge : ShaclShape -> CustomConstraint list -> ShaclShape`
- [x] T033 Implement additive constraint merging (Pattern, MinInclusive, MaxInclusive, MinLength, MaxLength, InValues)
- [x] T034 Implement conflict detection: custom minCount 0 on required field, contradictory InValues, etc.
- [x] T035 Implement SPARQL-based cross-field constraint support (delegating to dotNetRdf SPARQL engine)
- [x] T036 Create `test/Frank.Validation.Tests/ShapeMergerTests.fs` with merge/conflict tests

### Implementation Notes
- Merge runs once at startup, not per request
- Custom `Pattern` on a field with auto-derived `Pattern`: both apply (AND semantics)
- Custom `InValues` on a field with auto-derived `InValues`: intersection (tighter constraint)
- Custom `MinCount 0` on a required field (auto-derived `MinCount 1`): raise `InvalidOperationException` with descriptive message
- SPARQL constraints: stored as `sh:sparql` on the NodeShape, evaluated by dotNetRdf's SPARQL engine during validation
- `CustomShaclConstraint` escape hatch: raw predicate URI + value for constraints not covered by `ConstraintKind`

### Parallel Opportunities
- T033 (additive merging) and T034 (conflict detection) can proceed in parallel
- T035 (SPARQL) is independent of T033/T034

### Dependencies
- Depends on WP01 (types), WP02 (shape derivation produces base shapes)

### Risks & Mitigations
- Conflict detection false positives: conservative detection -- only flag direct contradictions, not semantic tensions
- SPARQL constraint complexity: limit to well-formed ASK queries; validate SPARQL syntax at startup
