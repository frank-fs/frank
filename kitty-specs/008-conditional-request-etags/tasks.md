## Work Package WP01: Core ETag Types & Interface (Priority: P0)

**Goal**: Define `IETagProvider`, `IETagProviderFactory`, `ETagMetadata`, `ETagFormat`, and `ETagComparison` modules in Frank core.
**Independent Test**: All types compile, ETagFormat and ETagComparison pass unit tests, strong ETag formatting conforms to RFC 9110.
**Prompt**: `tasks/WP01-core-etag-types-and-interface.md`
**Estimated Size**: ~400 lines
**Requirement Refs**: FR-007, FR-011, FR-012, FR-013, FR-014

### Included Subtasks
- [x] T001 Create `src/Frank/ETag.fs` with `IETagProvider` and `IETagProviderFactory` interfaces
- [x] T002 Define `ETagMetadata` sealed class (endpoint metadata marker with ProviderKey and ResolveInstanceId)
- [x] T003 Implement `ETagFormat` module (quote, unquote, isWeak, computeFromBytes with SHA-256 truncated to 128 bits)
- [x] T004 Implement `ETagComparison` module (strongMatch, parseIfNoneMatch, parseIfMatch, anyMatch)
- [x] T005 Create `test/Frank.Tests/ETagTests.fs` with unit tests for ETagFormat and ETagComparison

### Implementation Notes
- All types live in a single file `ETag.fs` in `src/Frank/`
- Add `ETag.fs` to `Frank.fsproj` compilation order (before Builder.fs)
- SHA-256 via `System.Security.Cryptography.SHA256` -- no external NuGet deps
- Strong ETags only: quoted-string format `"<32 hex chars>"`
- `IETagProvider` is non-generic (middleware does not know state/context types)
- `ETagMetadata` uses `HttpContext -> string` for instance ID resolution (same pattern as `StateMachineMetadata`)

### Dependencies
- None (starting package)

### Risks & Mitigations
- ETag header parsing edge cases (whitespace, empty values): comprehensive test coverage
- `computeFromBytes` must be deterministic across .NET versions: SHA-256 is standardized

## Work Package WP02: MailboxProcessor ETag Cache (Priority: P0)

**Goal**: Implement `ETagCache` with MailboxProcessor-backed concurrent cache, LRU eviction, and IDisposable lifecycle.
**Independent Test**: Cache can get/set/invalidate ETags, LRU eviction works at capacity, concurrent access is serialized, dispose cleans up.
**Prompt**: `tasks/WP02-etag-cache.md`
**Estimated Size**: ~350 lines
**Requirement Refs**: FR-009, FR-015

### Included Subtasks
- [x] T006 Create `src/Frank/ETagCache.fs` with `ETagCacheMessage` DU and `CacheEntry` record
- [x] T007 Implement `ETagCache` type with MailboxProcessor (GetETag, SetETag, Invalidate, InvalidateAll, GetStats)
- [x] T008 Implement LRU eviction when cache exceeds configurable maxEntries (default 10,000)
- [x] T009 Implement `IDisposable` on `ETagCache` with MailboxProcessor cleanup
- [x] T010 Create `test/Frank.Tests/ETagCacheTests.fs` with concurrency and eviction tests

### Implementation Notes
- Add `ETagCache.fs` to `Frank.fsproj` after `ETag.fs`
- `GetETag` uses `AsyncReplyChannel` (blocking); `SetETag`/`Invalidate` are fire-and-forget
- `LastAccessed` updated on both get and set operations
- `ILogger<ETagCache>` for error logging (Constitution principle VII)
- MailboxProcessor `Error` event wired to ILogger
- Register as singleton in DI via `IServiceCollection.AddETagCache()`

### Dependencies
- Depends on WP01 (uses ETag string type from ETagFormat)

### Risks & Mitigations
- MailboxProcessor disposal: `Stop` message drains pending operations before completing
- LRU eviction overhead: only scan on `SetETag` when at capacity; O(n) scan is acceptable for default 10k entries

## Work Package WP03: Conditional Request Middleware (Priority: P0)

**Goal**: Implement `ConditionalRequestMiddleware` that evaluates `If-None-Match` (304) and `If-Match` (412) headers, and a `useConditionalRequests` plug for WebHostBuilder.
**Independent Test**: Middleware returns 304 for matching GET, 412 for stale mutations, passes through non-ETag resources and requests without conditional headers.
**Prompt**: `tasks/WP03-conditional-request-middleware.md`
**Estimated Size**: ~500 lines
**Requirement Refs**: FR-001, FR-003, FR-004, FR-005, FR-006, FR-011, FR-012, FR-013, FR-015

### Included Subtasks
- [x] T011 Create `src/Frank/ConditionalRequestMiddleware.fs` with middleware skeleton (endpoint metadata discovery, pass-through for non-ETag resources)
- [x] T012 Implement If-None-Match evaluation: GET/HEAD with matching ETag returns 304 Not Modified
- [x] T013 Implement If-Match evaluation: POST/PUT/DELETE with non-matching ETag returns 412 Precondition Failed
- [x] T014 Implement wildcard `*` handling for both If-None-Match and If-Match per RFC 9110
- [x] T015 Implement ETag header setting on successful responses and cache invalidation after mutations. Cache invalidation triggers after observing a 2xx response status from the handler, not before handler execution
- [x] T016 Add `useConditionalRequests` plug registration and `AddETagCache` DI extension
- [x] T017 Create `test/Frank.Tests/ConditionalRequestTests.fs` with TestHost integration tests

### Implementation Notes
- Add `ConditionalRequestMiddleware.fs` to `Frank.fsproj` after `ETagCache.fs`
- Middleware runs after routing (needs endpoint metadata) but before handler execution
- On cache miss: compute ETag via `IETagProvider`, store in cache, then evaluate
- After successful mutation (2xx response), invalidate cache entry to force recomputation
- Set `ETag` header on 200 responses for ETag-enabled resources
- Set `ETag` header on 304 responses (client needs it for subsequent requests)
- `useConditionalRequests` registers middleware via `plug` (after `UseRouting`)
- Multiple ETags in If-None-Match: comma-separated per RFC 9110, any match triggers 304

### Dependencies
- Depends on WP01 (ETagMetadata, ETagComparison, ETagFormat) and WP02 (ETagCache)

### Risks & Mitigations
- Middleware ordering with Auth/Statecharts: document recommended order in plug registration
- Race between ETag check and handler: MailboxProcessor serializes cache access; inherent to optimistic concurrency

## Work Package WP04: StatechartETagProvider (Priority: P1)

**Goal**: Implement default `IETagProvider` for statechart-backed resources in `Frank.Statecharts`, with SHA-256 hashing of `('State * 'Context)` pairs and cache invalidation on state transitions.
**Independent Test**: Provider computes deterministic ETags from state, ETags change on transitions, cache is invalidated on state change.
**Prompt**: `tasks/WP04-statechart-etag-provider.md`
**Estimated Size**: ~350 lines
**Requirement Refs**: FR-002, FR-008, FR-010, FR-015

### Included Subtasks
- [x] T018 Create `src/Frank.Statecharts/StatechartETagProvider.fs` implementing `IETagProvider`
- [x] T019 Implement SHA-256 hashing: serialize state via `string`, context via user-supplied `'Context -> byte[]`, concatenate, hash, truncate to 128 bits
- [ ] T019b Unit test: same ('State * 'Context) pair produces identical ETag; different pairs produce different ETags
- [x] T020 Implement `IETagProviderFactory` for statechart resources (resolves provider from endpoint StateMachineMetadata)
- [x] T021 Implement middleware-driven cache invalidation: after the handler returns a 2xx response for a mutation (POST/PUT/DELETE), the middleware sends an `InvalidateETag` message to the `ETagCache` (consistent with T015 in WP03). Note: `IStateMachineStore.Subscribe` is per-instance only and cannot be used for global cache invalidation.
- [x] T022 Add DI registration helper: `IServiceCollection.AddStatechartETagProvider<'State, 'Context>(contextSerializer)`

### Implementation Notes
- `StatechartETagProvider.fs` goes in `src/Frank.Statecharts/` (depends on `IStateMachineStore`)
- Add to `Frank.Statecharts.fsproj` compilation order (after existing files)
- Provider is generic `<'State, 'Context>` but implements non-generic `IETagProvider`
- `contextSerializer: 'Context -> byte[]` is provided at registration time
- State serialization: `System.Text.Encoding.UTF8.GetBytes(string state)` for DU states
- Hash: `SHA256.Create().ComputeHash(combined) |> Array.take 16 |> hex-encode |> ETagFormat.quote`
- Cache invalidation: middleware-driven (after 2xx mutation response, middleware posts `InvalidateETag` to cache); no store-level subscription needed
- `IStateMachineStore.Subscribe` is per-instance only and cannot provide global cache invalidation

### Dependencies
- Depends on WP01 (IETagProvider, ETagFormat) and WP02 (ETagCache for invalidation)
- Depends on Frank.Statecharts types (IStateMachineStore, StateMachineMetadata) -- already exist

### Risks & Mitigations
- Generic type erasure at metadata level: use boxed/erased interface (same pattern as `StateMachineMetadata.Machine: obj`)
- `string state` representation stability: DU case names are stable in F# -- safe to use

## Work Package WP05: Integration Tests & End-to-End Validation (Priority: P1)

**Goal**: End-to-end integration tests validating the full conditional request pipeline with both statechart and custom provider resources via TestHost.
**Independent Test**: Complete request lifecycle works: ETag generation, 304 Not Modified, 412 Precondition Failed, wildcard handling, multi-ETag If-None-Match, custom providers.
**Prompt**: `tasks/WP05-integration-tests.md`
**Estimated Size**: ~450 lines
**Requirement Refs**: FR-001, FR-002, FR-003, FR-004, FR-005, FR-006, FR-007, FR-008, FR-009, FR-010, FR-011, FR-012, FR-013, FR-014, FR-015

### Included Subtasks
- [x] T023 Create test fixture: simplified statechart resource (2-3 states) with ETag middleware in TestHost
- [x] T024 Test User Story 1: GET responses include ETag headers, ETag changes on state transition, initial state has ETag
- [ ] T025 Test User Story 2: conditional GET with matching If-None-Match returns 304, changed state returns 200 with new ETag, multiple ETags, wildcard
- [ ] T026 Test User Story 3: If-Match on mutations -- matching proceeds, non-matching returns 412, wildcard, no If-Match proceeds normally
- [ ] T027 Test User Story 4: custom IETagProvider on plain resource -- ETags generated, conditional requests work identically
- [ ] T028 Test edge cases: resource without IETagProvider unaffected, DELETE responses, concurrent clients (optimistic concurrency scenario)
- [ ] T028b Benchmark ETag overhead via BenchmarkDotNet — verify <1ms for typical state sizes

### Implementation Notes
- Tests go in `test/Frank.Tests/ConditionalRequestIntegrationTests.fs` (separate from unit tests in WP03)
- Use ASP.NET Core TestHost (`Microsoft.AspNetCore.TestHost`) matching existing Frank.Tests patterns
- Test fixture: simple 2-state machine (e.g., Active/Completed) with POST to transition
- Custom provider test: implement a simple `IETagProvider` that returns a version-based ETag
- Optimistic concurrency test: two clients GET (same ETag), client A POSTs (succeeds), client B POSTs with old ETag (412)
- Verify ETag format: quoted string, 32 hex chars, changes on state transition

### Dependencies
- Depends on WP03 (middleware) and WP04 (StatechartETagProvider)

### Risks & Mitigations
- TestHost setup complexity: reference existing Frank.Tests patterns for middleware registration
- Async timing in concurrent tests: use deterministic sequencing rather than actual concurrency
