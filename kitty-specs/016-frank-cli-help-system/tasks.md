## Work Package WP01: Foundation Types and Utilities (Priority: P0)

**Goal**: Define all help system types (CommandHelp, WorkflowPosition, HelpTopic, ProjectStatus, ExtractionStatus, ArtifactStatus, RecommendedAction, HelpLookupResult, StalenessResult) and implement the FuzzyMatch utility module. These types are the data backbone for all subsequent work packages.
**Independent Test**: `dotnet build src/Frank.Cli.Core/Frank.Cli.Core.fsproj` succeeds with the new files compiled.
**Prompt**: `/tasks/WP01-foundation-types-and-utilities.md`
**Estimated Prompt Size**: ~350 lines

### Included Subtasks
- [x] T001 Create `src/Frank.Cli.Core/Help/HelpTypes.fs` with CommandExample, WorkflowPosition, CommandHelp, HelpTopic, HelpLookupResult types
- [x] T002 Create `src/Frank.Cli.Core/Help/FuzzyMatch.fs` with Levenshtein distance and suggestion ranking
- [x] T003 Create `src/Frank.Cli.Core/Commands/StalenessChecker.fs` with StalenessResult type, computeFileHash, and checkStaleness extracted from ValidateCommand
- [x] T004 Add ProjectStatus types (ExtractionStatus, ArtifactStatus, RecommendedAction, ProjectStatus record) to HelpTypes.fs
- [x] T005 Update `src/Frank.Cli.Core/Frank.Cli.Core.fsproj` -- add Compile entries for HelpTypes.fs, FuzzyMatch.fs, StalenessChecker.fs in correct dependency order
- [x] T006 Refactor `src/Frank.Cli.Core/Commands/ValidateCommand.fs` to call StalenessChecker instead of local computeFileHash/checkStaleness

### Implementation Notes
- HelpTypes.fs must be compiled before FuzzyMatch.fs and StalenessChecker.fs.
- StalenessChecker.fs must appear before ValidateCommand.fs and StatusCommand.fs in the compile order.
- The ValidateCommand refactoring replaces private `computeFileHash` (lines 124-128) and `checkStaleness` (lines 130-164) with calls to the shared StalenessChecker module. The ValidateCommand then maps `StalenessResult.Stale` to a `ValidationIssue`.
- FuzzyMatch uses simple Levenshtein distance + prefix matching for 9 candidates (7 commands + 2 topics).

### Parallel Opportunities
- T001 and T002 can be developed in parallel (different files, no mutual dependency).
- T003 can proceed in parallel with T001/T002 (different module).

### Dependencies
- None (starting package).

### Risks & Mitigations
- ValidateCommand refactoring could break existing tests -- run `dotnet test` for Frank.Cli.Core.Tests after refactoring to confirm ValidateCommandTests still pass.
- StalenessChecker compile order must be before ValidateCommand.fs -- verify by building after .fsproj update.

## Work Package WP03: Help Subcommand Logic (Priority: P0)

**Goal**: Implement the help subcommand resolution logic: looking up commands, topics, providing fuzzy "did you mean?" suggestions, and listing all commands/topics when no argument is given.
**Independent Test**: `dotnet build src/Frank.Cli.Core/Frank.Cli.Core.fsproj` succeeds. HelpSubcommand functions can be unit-tested against HelpContent data.
**Prompt**: `/tasks/WP03-help-subcommand-logic.md`
**Estimated Prompt Size**: ~300 lines

### Included Subtasks
- [ ] T013 Create `src/Frank.Cli.Core/Help/HelpSubcommand.fs` with help argument resolution logic
- [ ] T014 Implement `resolve` function: given a string argument, return HelpLookupResult (CommandMatch, TopicMatch, or NoMatch with suggestions)
- [ ] T015 Implement "did you mean?" suggestion logic using FuzzyMatch (threshold: distance <= 3 or <= 50% of input length, plus prefix matching)
- [ ] T016 Implement `listAll` function: return all commands and topics for the no-argument case
- [ ] T017 Update `src/Frank.Cli.Core/Frank.Cli.Core.fsproj` -- add Compile entry for HelpSubcommand.fs

### Implementation Notes
- Commands take priority over topics when a name matches both (edge case from spec).
- The resolve function should check exact match first (commands then topics), then fall back to fuzzy matching.
- Suggestion results should be sorted by distance (closest first).
- The no-argument `listAll` case returns command summaries and topic summaries for the index view.

### Parallel Opportunities
- T013-T016 are in a single file and must be sequential within this WP.

### Dependencies
- Depends on WP01 (HelpTypes, FuzzyMatch) and WP02 (HelpContent).

### Risks & Mitigations
- Fuzzy match threshold tuning -- with only 9 candidates, the threshold of distance <= 3 should be sufficient. Test with common typos: "comiple" -> "compile", "extact" -> "extract".

- JSON output must be valid and parseable -- use Utf8JsonWriter which guarantees well-formed JSON.

## Work Package WP05: System.CommandLine Integration and Program.fs Wiring (Priority: P1)

**Goal**: Create the custom HelpAction wrapper that appends WORKFLOW/EXAMPLES/CONTEXT sections to standard --help output, add the System.CommandLine package dependency, and wire everything into Program.fs (register help and status commands, add --context global option, attach custom HelpAction).
**Independent Test**: `dotnet run --project src/Frank.Cli -- extract --help` shows WORKFLOW and EXAMPLES sections. `dotnet run --project src/Frank.Cli -- help` shows command/topic listing. `dotnet run --project src/Frank.Cli -- status --project <path>` reports project state.
**Prompt**: `/tasks/WP05-scl-integration-and-wiring.md`
**Estimated Prompt Size**: ~450 lines

### Included Subtasks
- [ ] T025 Create `src/Frank.Cli.Core/Help/HelpAction.fs` with custom SynchronousCommandLineAction that wraps default HelpAction
- [ ] T026 Update `src/Frank.Cli.Core/Frank.Cli.Core.fsproj` -- add System.CommandLine 2.0.3 PackageReference and HelpAction.fs Compile entry
- [ ] T027 Update `src/Frank.Cli/Program.fs` -- add `--context` global option on RootCommand
- [ ] T028 Update `src/Frank.Cli/Program.fs` -- register `status` subcommand with --project and --format options
- [ ] T029 Update `src/Frank.Cli/Program.fs` -- register `help` subcommand with string argument and --format option
- [ ] T030 Update `src/Frank.Cli/Program.fs` -- find HelpOption on root command and replace its action with custom HelpAction

### Implementation Notes
- HelpAction.fs uses the `HelpOption.Action` wrapper pattern documented in research.md (R1).
- The custom action calls `HelpAction.Invoke(parseResult)` first for standard output, then appends WORKFLOW/EXAMPLES (and optionally CONTEXT) via HelpRenderer.
- The --context option is `Option<bool>("--context")` added as a global option on the root command.
- StatusCommand wiring: `--project` is required string option, `--format` defaults to "text".
- Help subcommand: takes an optional `Argument<string>` (the command or topic name), `--format` defaults to "text".
- Error handling: status command returns exit code 1 on missing project; help command returns 1 for unknown arguments.

### Parallel Opportunities
- T025 (HelpAction.fs) can be developed independently of T027-T030 (Program.fs changes).
- T028 and T029 (status and help registration) touch different sections of Program.fs but must be applied sequentially.

### Dependencies
- Depends on WP01 (types), WP02 (content, StatusCommand), WP03 (HelpSubcommand), WP04 (rendering).

### Risks & Mitigations
- System.CommandLine HelpAction API -- the wrapper pattern is documented and verified in research.md R1. If the API has changed, fall back to `HelpBuilder.CustomizeLayout`.
- Compile order matters: HelpAction.fs must be after HelpRenderer.fs in the .fsproj Compile list.
- Program.fs is the most complex change -- test manually with all commands from quickstart.md build/test section.
