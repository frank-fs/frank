# Implementation Plan: SCXML Parser and Generator

**Branch**: `018-scxml-parser-generator` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `kitty-specs/018-scxml-parser-generator/spec.md`
**GitHub Issue**: #98

## Summary

Bidirectional W3C SCXML support for the Frank.Statecharts spec pipeline. A parser reads SCXML XML documents (using `System.Xml.Linq`) into SCXML-specific parse types, and a generator produces valid W3C SCXML XML from those types. A separate mapper module converts between the SCXML-specific types and the shared AST (spec 020). The parser, generator, and their tests are independently buildable; only the mapper is blocked on spec 020 landing.

## Technical Context

**Language/Version**: F# 8.0+ targeting .NET 8.0/9.0/10.0 (multi-targeting)
**Primary Dependencies**: `System.Xml.Linq` (in-framework), `System.Xml` (for `IXmlLineInfo`, `XmlException`)
**Storage**: N/A (stateless parser/generator library)
**Testing**: Expecto 10.2.3 + YoloDev.Expecto.TestSdk 0.14.3 + Microsoft.NET.Test.Sdk 17.14.1
**Target Platform**: .NET multi-target (net8.0;net9.0;net10.0)
**Project Type**: Library (internal modules within `Frank.Statecharts`)
**Performance Goals**: 100+ state SCXML documents parse without measurable degradation (SC-007)
**Constraints**: No additional NuGet dependencies; `System.Xml.Linq` only
**Scale/Scope**: 9 SCXML element types in scope; ~6 source files for parser/generator/types/mapper

## Planning Answers

| # | Question | Answer |
|---|----------|--------|
| PQ1 | How to handle shared AST dependency (spec 020)? | Define SCXML-specific parse types for parsing. Separate mapper converts to shared AST. Parser, generator, and tests are independently buildable. Only the mapper WP is blocked on spec 020. |
| PQ2 | Where to store state-scoped `<datamodel>`/`<data>` entries? | As a field on the SCXML state node (`DataEntry list` per state), mirroring XML structure. |
| PQ3 | How to handle XML position tracking for errors? | `XDocument.Load`/`Parse` with `LoadOptions.SetLineInfo` for happy path (cast to `IXmlLineInfo`). Catch `XmlException` for malformed XML (carries line/column natively). |

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Resource-Oriented Design | N/A | This is a parser/generator library, not an HTTP resource definition. No resource-oriented API surface. |
| II. Idiomatic F# | PASS | Discriminated unions for SCXML node types, `option` types for optional attributes, `Result` for parse outcomes, immutable records throughout. |
| III. Library, Not Framework | PASS | Pure library: parser accepts XML input, returns typed results. No middleware, no runtime hooks, no opinions beyond SCXML parsing. |
| IV. ASP.NET Core Native | N/A | No ASP.NET Core surface. Library-level XML parsing only. |
| V. Performance Parity | PASS | `System.Xml.Linq` is the standard .NET XML stack. No custom allocations in hot paths. 100+ state benchmark planned. |
| VI. Resource Disposal Discipline | PASS | `XDocument.Parse` for string input (no disposable). For `TextReader`/`Stream` overloads, callers own the resource. `use` bindings for any internal `StringReader` usage. |
| VII. No Silent Exception Swallowing | PASS | `XmlException` is caught and converted to structured `ParseError` with line/column. No bare `with _ ->` handlers. |
| VIII. No Duplicated Logic Across Modules | PASS | Namespace resolution helper, element-to-node mapping, and attribute extraction are in shared helper functions within the `Scxml` module. No duplication with WSD parser (different parsing technology). |
