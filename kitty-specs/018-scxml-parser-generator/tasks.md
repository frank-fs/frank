# Work Packages: SCXML Parser and Generator

**Inputs**: Design documents from `/kitty-specs/018-scxml-parser-generator/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/, quickstart.md

**Tests**: Testing is included as explicit work packages (WP05, WP06) since the spec defines acceptance scenarios and success criteria requiring test coverage.

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package must be independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `/tasks/` generated by `/spec-kitty.tasks`. Treat this file as the high-level checklist; keep deep implementation detail inside the prompt files.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- Include precise file paths or modules.

## Path Conventions
- **Source**: `src/Frank.Statecharts/Scxml/`
- **Tests**: `test/Frank.Statecharts.Tests/Scxml/`

## Work Package WP02: SCXML Parser -- Core Parsing (Priority: P1)

**Goal**: Implement the SCXML parser core: XML loading with namespace handling, `<scxml>` root parsing, `<state>`/`<final>`/`<parallel>` element parsing, `<transition>` parsing, and initial state inference. After this WP, the parser can handle User Story 1 (parse states, transitions, events, guards).
**Independent Test**: `dotnet build src/Frank.Statecharts` succeeds; `parseString` on a simple SCXML document with states and transitions returns a populated `ScxmlDocument`.
**Prompt**: `/tasks/WP02-scxml-parser-core.md`
**Estimated prompt size**: ~450 lines

### Included Subtasks
- [ ] T003 Create `src/Frank.Statecharts/Scxml/Parser.fs` with SCXML namespace constant, `XDocument` loading helper (`LoadOptions.SetLineInfo`), and `extractPosition` helper (via `IXmlLineInfo` cast)
- [ ] T004 Implement `<scxml>` root element parsing: extract `initial`, `name`, `datamodel`, `binding` attributes, validate root element namespace
- [ ] T005 Implement `<state>`, `<final>`, `<parallel>` element parsing with `ScxmlStateKind` derivation (Simple/Compound/Parallel/Final), recursive child state parsing
- [ ] T006 Implement `<transition>` parsing: extract `event`, `cond`, `target` (space-split into list), `type` (default External) attributes
- [ ] T009 Implement initial state inference (when `<scxml>` has no `initial` attribute, use first child state ID per W3C section 3.2) and compound state detection (state with child states -> Compound)
- [ ] T012 Update `Frank.Statecharts.fsproj` to add `Scxml/Parser.fs` compile entry after `Scxml/Types.fs`

### Implementation Notes
- Module: `module internal Frank.Statecharts.Scxml.Parser`
- Use `XNamespace.Get("http://www.w3.org/2005/07/scxml")` for namespace-qualified element lookups
- Parser returns `ScxmlParseResult` with `Document`, `Errors`, and `Warnings` lists
- Recursive `parseState` function handles the tree structure
- Stub out `<datamodel>`, `<history>`, `<invoke>` parsing as empty lists (WP03 fills these in)

### Parallel Opportunities
- T003/T004/T005/T006 form a sequential chain within the module, but the file can be written in a single pass.

### Dependencies
- Depends on WP01 (requires `Scxml/Types.fs`).

### Risks & Mitigations
- Namespace resolution edge cases: test both default and prefixed namespaces.
- Recursive state parsing for deep hierarchies: ensure tail-recursion or bounded depth.

## Work Package WP03: SCXML Parser -- Data Model, History, Invoke, Errors, and Overloads (Priority: P1)

**Goal**: Complete the parser with `<datamodel>`/`<data>` parsing, `<history>` and `<invoke>` parsing, structured error handling (malformed XML, structural validation), and `parseReader`/`parseStream` overloads. After this WP, the parser fully satisfies User Stories 1, 3, and 4.
**Independent Test**: `dotnet build src/Frank.Statecharts` succeeds; parser handles data model entries, history/invoke elements, malformed XML input, and Stream/TextReader inputs.
**Prompt**: `/tasks/WP03-scxml-parser-advanced.md`
**Estimated prompt size**: ~400 lines

### Included Subtasks
- [ ] T007 Implement `<datamodel>`/`<data>` parsing: support both `expr` attribute and child text content forms, state-scoped data entries (data on `ScxmlState.DataEntries`) vs. top-level (on `ScxmlDocument.DataEntries`)
- [ ] T008 Implement `<history>` parsing (id, type defaulting to Shallow, optional child `<transition>` as default target) and `<invoke>` parsing (type, src, id attributes)
- [ ] T010 Implement structured error handling: catch `System.Xml.XmlException` for malformed XML (extract LineNumber/LinePosition into `ParseError`), emit warnings for unknown/out-of-scope elements, validate structural issues (e.g., invalid `<history type="...">` value)
- [ ] T011 Implement `parseReader : TextReader -> ScxmlParseResult` and `parseStream : Stream -> ScxmlParseResult` overloads using `XDocument.Load` with `LoadOptions.SetLineInfo`

### Implementation Notes
- `<data>` elements: `expr` attribute takes precedence over child text content (per parser-api.md contract)
- Out-of-scope elements (`<onentry>`, `<onexit>`, `<script>`, etc.) should be silently skipped with a warning
- Error handling must not use bare `with _ ->` (constitution principle VII)
- `parseReader` and `parseStream` share the core parsing logic; only the `XDocument` loading differs

### Parallel Opportunities
- T007 and T008 touch different element types and could be implemented concurrently in theory, but they modify the same function flow.

### Dependencies
- Depends on WP02 (builds on the parser skeleton from WP02).

### Risks & Mitigations
- `<data>` with child content: must handle mixed content (text interspersed with child elements) gracefully.
- `XmlException` properties: ensure `LineNumber` and `LinePosition` are 1-based (they are in .NET).

## Work Package WP04: SCXML Generator (Priority: P1)

**Goal**: Implement the SCXML XML generator that produces valid W3C SCXML XML from `ScxmlDocument` types. Covers all nine in-scope element types. After this WP, User Story 2 is fully satisfied.
**Independent Test**: `dotnet build src/Frank.Statecharts` succeeds; `generate` on a programmatically constructed `ScxmlDocument` produces valid, re-parseable SCXML XML.
**Prompt**: `/tasks/WP04-scxml-generator.md`
**Estimated prompt size**: ~450 lines

### Included Subtasks
- [x] T013 Create `src/Frank.Statecharts/Scxml/Generator.fs` with module declaration, SCXML namespace/version constants, and `generate` function scaffolding using `XDocument`/`XElement`
- [x] T014 Implement state element generation: emit `<state>`, `<final>`, or `<parallel>` based on `ScxmlStateKind`, with `id` and `initial` attributes, recursive child state generation for compound/parallel states
- [x] T015 Implement transition element generation: emit `<transition>` with `event`, `cond`, `target` (joined with spaces for multi-target), and `type` attributes (omit `type` when External/default)
- [ ] T016 Implement datamodel/data element generation: emit `<datamodel>` with `<data id="..." expr="..."/>` children; entries with no expression emit `<data id="..."/>`
- [ ] T017 Implement history and invoke element generation: emit `<history id="..." type="...">` with optional child `<transition>`, emit `<invoke>` with `type`, `src`, `id` attributes
- [ ] T018 Implement `generateTo : TextWriter -> ScxmlDocument -> unit` overload that writes directly to a `TextWriter`
- [ ] T019 Update `Frank.Statecharts.fsproj` to add `Scxml/Generator.fs` compile entry after `Scxml/Parser.fs`

### Implementation Notes
- Module: `module internal Frank.Statecharts.Scxml.Generator`
- Root element: always `<scxml xmlns="http://www.w3.org/2005/07/scxml" version="1.0">`
- Optional attributes on `<scxml>` (`initial`, `name`, `datamodel`, `binding`) emitted only when `Some`
- Canonical attribute order: `id`, then `initial`/`type`/other, then `event`, `cond`, `target` (per generator-api.md)
- `generate` returns indented XML string; `generateTo` writes to `TextWriter`
- Use `XDocument.Save` with `SaveOptions.None` for indented output

### Parallel Opportunities
- T014 (states), T015 (transitions), T016 (datamodel), T017 (history/invoke) touch different element types and could be developed concurrently, though they compose into the same recursive function.

### Dependencies
- Depends on WP01 (requires `Scxml/Types.fs`).
- Can be developed in parallel with WP02 and WP03 (both only depend on WP01).

### Risks & Mitigations
- XML namespace declarations must appear only on the root element (avoid duplicate `xmlns` on child elements).
- `XElement` constructor with `XNamespace` automatically handles namespace inheritance for child elements.

## Dependency & Execution Summary

- **Sequence**: WP01 -> WP02, WP04 (parallel) -> WP03 -> WP05, WP06 (parallel after prerequisites)
- **Parallelization**:
  - WP02 (parser core) and WP04 (generator) can proceed in parallel after WP01
