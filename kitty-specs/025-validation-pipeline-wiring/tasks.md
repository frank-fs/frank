# Work Packages: Validation Pipeline End-to-End Wiring

**Inputs**: Design documents from `/kitty-specs/025-validation-pipeline-wiring/`
**Prerequisites**: plan.md (required), spec.md (user stories), data-model.md, quickstart.md

**Tests**: Yes -- end-to-end integration tests and unit tests are explicitly required by the spec (User Story 5, SC-001 through SC-007).

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package must be independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `/tasks/` generated by `/spec-kitty.tasks`. Treat this file as the high-level checklist; keep deep implementation detail inside the prompt files.

## Work Package WP01: Pipeline Types & Module Foundation (Priority: P0)

**Goal**: Add the three new types (`PipelineError`, `FormatParseResult`, `PipelineResult`) to `Validation/Types.fs`, create `Validation/Pipeline.fs` with the `parserFor` dispatcher and `validateSources`/`validateSourcesWithRules` functions, and wire it into the `.fsproj` compile order.
**Independent Test**: `dotnet build` succeeds with the new types and Pipeline module. The module is public and visible from external assemblies.
**Prompt**: `tasks/WP01-pipeline-types-and-module.md`
**Estimated Size**: ~350 lines

### Included Subtasks
- [x] T001 Add `PipelineError` discriminated union to `src/Frank.Statecharts/Validation/Types.fs`
- [x] T002 Add `FormatParseResult` record type to `src/Frank.Statecharts/Validation/Types.fs`
- [x] T003 Add `PipelineResult` record type to `src/Frank.Statecharts/Validation/Types.fs`
- [x] T004 Create `src/Frank.Statecharts/Validation/Pipeline.fs` with `parserFor` private function
- [x] T005 Implement `validateSourcesWithRules` public function in `Pipeline.fs`
- [x] T006 Implement `validateSources` as shorthand calling `validateSourcesWithRules []`
- [x] T007 Add `Compile Include` for `Validation/Pipeline.fs` in `Frank.Statecharts.fsproj` after `Validation/Validator.fs`

### Implementation Notes
- Types must be added below the existing `ValidationRule` type in `Types.fs` to avoid forward-reference issues.
- `Pipeline.fs` namespace is `Frank.Statecharts.Validation` and the module must be public (not `internal`), per FR-013.
- The `parserFor` function must handle the current parser interfaces: WSD and smcat return `Ast.ParseResult` directly; SCXML returns `ScxmlParseResult` (needs `Scxml.Mapper.toStatechartDocument`); ALPS returns `Result<AlpsDocument, AlpsParseError list>` (needs `Alps.Mapper.toStatechartDocument` + error wrapping). The spec says "post-migration" but implementation must work against the current codebase.
- `Pipeline.fs` must appear after all parser and mapper files in the .fsproj compile order but can be placed immediately after `Validation/Validator.fs`.
- Input validation: reject duplicate `FormatTag` values (FR-010), handle empty input (FR-011), report unsupported formats (FR-012).

### Parallel Opportunities
- T001/T002/T003 are tightly coupled (same file) -- sequential within the file but independent from T004-T007.
- T004/T005/T006 are sequential within `Pipeline.fs`.

### Dependencies
- None (starting package).

### Risks & Mitigations
- **Parser interface mismatch**: SCXML and ALPS parsers do NOT yet return `Ast.ParseResult` directly -- they return format-specific types. The Pipeline must adapt via Mapper modules. The spec assumes post-migration but we need to handle the current interface until #113/#114/#115 complete.
- **Compile ordering**: F# is sensitive to file ordering. Pipeline.fs must see all parser modules. Mitigate by placing it right after `Validation/Validator.fs` in the fsproj (which already comes after all parsers).

## Work Package WP02: Pipeline Unit Tests (Priority: P1)

**Goal**: Create `PipelineTests.fs` with unit tests covering all pipeline dispatch logic, edge cases (empty input, duplicate formats, unsupported formats), parse error handling, single-format validation, and custom rules via `validateSourcesWithRules`.
**Independent Test**: `dotnet test --filter "Validation.Pipeline"` passes with all unit tests green.
**Prompt**: `tasks/WP02-pipeline-unit-tests.md`
**Estimated Size**: ~400 lines

### Included Subtasks
- [x] T008 Create `test/Frank.Statecharts.Tests/Validation/PipelineTests.fs` with test module and helpers
- [x] T009 Test: empty input returns valid PipelineResult with empty report (FR-011)
- [x] T010 Test: duplicate format tags return `DuplicateFormat` pipeline error (FR-010)
- [x] T011 Test: unsupported format tag (XState) returns `UnsupportedFormat` error (FR-012)
- [x] T012 Test: single format runs self-consistency, cross-format rules skipped (User Story 3)
- [x] T013 Test: two consistent formats produce zero validation failures (User Story 1, scenario 1)
- [x] T014 Test: parse errors included in `FormatParseResult.Errors` (FR-008, User Story 2)
- [x] T015 Test: `validateSourcesWithRules` includes custom rules (FR-014)
- [x] T016 Add `Compile Include` for `Validation/PipelineTests.fs` in `Frank.Statecharts.Tests.fsproj`

### Implementation Notes
- Tests use Expecto (`testList`, `test`, `Expect.*`) matching the project's test style.
- For parse error tests (T014), provide intentionally malformed source text (e.g., WSD with syntax errors) and verify errors appear in the `FormatParseResult`.
- For single-format test (T012), verify `Report.TotalSkipped > 0` because cross-format rules require pairs.
- Place `PipelineTests.fs` after existing `Validation/IntegrationTests.fs` in the test fsproj.

### Parallel Opportunities
- T009 through T015 are independent test cases and can be written in any order.

### Dependencies
- Depends on WP01 (Pipeline types and module must exist).

### Risks & Mitigations
- **Parser availability**: If parsers are `internal`, tests that call `Pipeline.validateSources` with real source text will exercise them transitively. The pipeline is the public API surface.
- **Test isolation**: Each test should construct its own input, not share mutable state.

## Work Package WP03: End-to-End Integration Tests (Priority: P1) -- MVP

**Goal**: Create `PipelineIntegrationTests.fs` with end-to-end tests using real tic-tac-toe format source text in WSD, smcat, SCXML, and ALPS. Verify zero failures for consistent inputs, correct mismatch detection for intentional differences, parse error handling with real malformed input, and performance (4 formats in under 2 seconds).
**Independent Test**: `dotnet test --filter "Validation.PipelineIntegration"` passes with all integration tests green.
**Prompt**: `tasks/WP03-e2e-integration-tests.md`
**Estimated Size**: ~450 lines

### Included Subtasks
- [ ] T017 Create `test/Frank.Statecharts.Tests/Validation/PipelineIntegrationTests.fs` with real format source constants
- [ ] T018 [P] Define tic-tac-toe WSD source text constant (states: idle, playerX, playerO, gameOver; events: start, move, win)
- [ ] T019 [P] Define tic-tac-toe smcat source text constant (same state machine)
- [ ] T020 [P] Define tic-tac-toe SCXML source text constant (same state machine)
- [ ] T021 [P] Define tic-tac-toe ALPS source text constant (same state machine)
- [ ] T022 Test: consistent 4-format sources produce zero validation failures (SC-001, User Story 5 scenario 1)
- [ ] T023 Test: state name mismatch detected (rename gameOver to finished in smcat) (SC-002, User Story 5 scenario 2)
- [ ] T024 Test: missing event detected (remove start event from ALPS) (User Story 5 scenario 3)
- [ ] T025 Test: parse error in one format still validates others (User Story 2, scenario 1)
- [ ] T026 Test: performance -- 4 formats under 2 seconds (SC-004)
- [ ] T027 Add `Compile Include` for `Validation/PipelineIntegrationTests.fs` in `Frank.Statecharts.Tests.fsproj`

### Implementation Notes
- Source text constants must produce parseable documents with the correct states and transitions. Use the quickstart.md examples as a starting point for WSD and smcat syntax. For SCXML, use valid `<scxml>` XML. For ALPS, use valid ALPS JSON.
- The existing `IntegrationTests.fs` uses hand-constructed AST. This WP is different -- it parses real format text through real parsers via the pipeline.
- For the mismatch test (T023), modify only the smcat source to rename `gameOver` to `finished` and verify the pipeline detects the disagreement.
- For the parse error test (T025), provide valid WSD + malformed SCXML (e.g., missing closing tag) and verify: SCXML parse errors in result, WSD artifact still validated.
- Performance test (T026): use a Stopwatch, assert < 2 seconds.

### Parallel Opportunities
- T018/T019/T020/T021 are independent source text definitions.
- T022-T026 are independent test cases.

### Dependencies
- Depends on WP01 (Pipeline module) and WP02 (unit tests should pass first to validate basic pipeline behavior before running integration tests).

### Risks & Mitigations
- **Parser-specific syntax quirks**: Real format text may trigger unexpected parser behavior. Use simple state machines that exercise core features without hitting edge cases in individual parsers.
- **ALPS format complexity**: ALPS is JSON-based with a different conceptual model. The ALPS source text must encode states and transitions in a way the ALPS parser can map to `StatechartDocument` states and transitions. Consult existing ALPS test fixtures in `test/Frank.Statecharts.Tests/Alps/` for syntax guidance.
- **SCXML/ALPS mapper dependency**: Until #113/#114/#115 complete, the pipeline must call mappers after parsing. Integration tests will verify this adapter logic works correctly end-to-end.
