## Work Package WP01: State Key Extraction (Priority: P1)

**Goal**: Replace fragile `state.ToString()` key derivation with `FSharpValue.PreComputeUnionTagReader` + `FSharpType.GetUnionCases` so parameterized DU cases (e.g., `Won "X"`, `Won "O"`) map to a single handler key (`"Won"`).
**Independent Test**: Define a state machine with parameterized DU states. Register one handler for a parameterized case. Verify both parameter variants match the same handler. Verify simple (non-parameterized) cases are unchanged. Build across all targets.
**Prompt**: `/tasks/WP01-state-key-extraction.md`
**Estimated Size**: ~450 lines

### Included Subtasks
- [ ] T001 Create `stateKey` function using `FSharpValue.PreComputeUnionTagReader` + `FSharpType.GetUnionCases` in `StatefulResourceBuilder.Run`
- [ ] T002 Add non-DU type fallback with `FSharpType.IsUnion` guard (falls back to `ToString()`)
- [ ] T003 Replace `state.ToString()` with `stateKey` at three call sites in `StatefulResourceBuilder.fs` (lines 176, 189, 250)
- [ ] T004 Change `InState` CE operation to use string keys (case names) instead of `'State` equality for `StateHandlerMap`
- [ ] T005 [P] Add tests for parameterized DU state matching in `StatefulResourceTests.fs`
- [ ] T006 [P] Add tests for simple DU backward compatibility (no regressions)
- [ ] T007 [P] Verify `StatechartETagProvider` is NOT affected (still uses `string state` for parameter-sensitive hashing)

### Implementation Notes
- The `stateKey` function is built once in `StatefulResourceBuilder.Run` and captured in closures
- `FSharpValue.PreComputeUnionTagReader(typeof<'State>)` returns `obj -> int` delegate (O(1) at runtime)
- `FSharpType.GetUnionCases(typeof<'State>)` returns `UnionCaseInfo[]` for case name lookup
- Three `.ToString()` call sites change to use `stateKey` function
- `StateHandlerMap` in the CE accumulator (`StatefulResourceSpec`) changes from `Map<'State, handlers>` to `Map<string, handlers>`
- Existing precedent: `src/Frank.LinkedData/Rdf/InstanceProjector.fs` line 94 uses `PreComputeUnionTagReader`

### Parallel Opportunities
- T005, T006, T007 (tests) can proceed in parallel after T001-T004 are complete

### Dependencies
- None (starting package)

### Risks & Mitigations
- `StateMetadata: Map<'State, StateInfo>` also uses `'State` as key -- must normalize to string keys when building `StateMachineMetadata` (covered in T004)
- Non-DU state types will fail `GetUnionCases` -- T002 adds the `FSharpType.IsUnion` guard

## Work Package WP02: Guard Discriminated Union (Priority: P2)

**Goal**: Replace the current `Guard` record type and `GuardContext` record with a `Guard` DU having `AccessControl` and `EventValidation` cases. Add `AccessControlContext` and `EventValidationContext` records. Update middleware to two-phase guard evaluation. Eliminate `Unchecked.defaultof<'E>`.
**Independent Test**: Define guards using both DU cases. Verify `AccessControl` guards run pre-handler (no event), `EventValidation` guards run post-handler (with actual event). No `Unchecked.defaultof` in codebase.
**Prompt**: `/tasks/WP02-guard-discriminated-union.md`
**Estimated Size**: ~550 lines

### Included Subtasks
- [ ] T008 Replace `Guard` record and `GuardContext` record with Guard DU + `AccessControlContext` + `EventValidationContext` in `Types.fs`
- [ ] T009 Update `StateMachine` record to use new `Guard<'S,'E,'C>` DU type (single `Guards` field, same name)
- [ ] T010 Update `evaluateGuards` closure in `StatefulResourceBuilder.fs` to split by DU pattern match (AccessControl only)
- [ ] T011 Create `evaluateEventGuards` closure in `StatefulResourceBuilder.fs` (EventValidation only, receives actual event)
- [ ] T012 Add `EvaluateEventGuards` field to `StateMachineMetadata` record
- [ ] T013 Update `Middleware.fs` to two-phase guard evaluation flow (step 3: AccessControl, step 5: EventValidation)
- [ ] T014 Update all existing tests referencing `Guard` record / `GuardContext` types
- [ ] T015 [P] Add tests for `AccessControl` guards (pre-handler, no event parameter)
- [ ] T016 [P] Add tests for `EventValidation` guards (post-handler, with actual event value)

### Implementation Notes
- The `Guard` DU replaces both the `Guard` record and the `GuardContext` record:
  ```fsharp
  type Guard<'State, 'Event, 'Context> =
      | AccessControl of name: string * predicate: (AccessControlContext<'State, 'Context> -> GuardResult)
      | EventValidation of name: string * predicate: (EventValidationContext<'State, 'Event, 'Context> -> GuardResult)
  ```
- New context records:
  ```fsharp
  type AccessControlContext<'State, 'Context> =
      { User: ClaimsPrincipal; CurrentState: 'State; Context: 'Context }
  type EventValidationContext<'State, 'Event, 'Context> =
      { User: ClaimsPrincipal; CurrentState: 'State; Event: 'Event; Context: 'Context }
  ```
- `StateMachine.Guards` stays as a single field with `Guard<'S,'E,'C> list`
- The builder splits guards at build time (in `Run`) into two closures: `evaluateGuards` (AccessControl) and `evaluateEventGuards` (EventValidation)
- Middleware flow changes from 5-step to 6-step (see research.md Decision 3)
- If an `EventValidation` guard blocks after the response has started, log a warning (same pattern as existing `TransitionAttemptResult.Blocked` handling)
- This is a source-breaking change -- acceptable for pre-1.0 library

### Parallel Opportunities
- T015 and T016 (new tests) can proceed in parallel after T008-T013 are complete

### Dependencies
- Depends on WP01 (shared `Types.fs` changes; WP01 stabilizes the `StateMachine` record shape before WP02 modifies it further)

### Risks & Mitigations
- Breaking change to `Guard` type requires updating all test files that construct guards -- T014 covers this comprehensively
- `EventValidation` guard blocking after response started -- handled with existing warning pattern (no new risk)
- `GuardContext` record removal -- must ensure no external consumers (internal only for pre-1.0)

## Work Package WP03: Actor Concurrency Validation & Documentation (Priority: P1)

**Goal**: Validate and document that the existing `IStateMachineStore` contract assumes actor-serialized access. Add concurrency tests proving the `MailboxProcessorStore` serializes concurrent state operations. Add documentation to `Store.fs`.
**Independent Test**: Run concurrent state mutation tests against `MailboxProcessorStore`. Verify sequential processing with no lost updates.
**Prompt**: `/tasks/WP03-actor-concurrency-validation.md`
**Estimated Size**: ~300 lines

### Included Subtasks
- [ ] T017 Add XML documentation to `IStateMachineStore` in `Store.fs` documenting actor-serialization contract requirement
- [ ] T018 Add concurrency serialization tests in `StoreTests.fs` (multiple concurrent `SetState` calls, verify sequential processing)
- [ ] T019 [P] Add test verifying no lost updates under concurrent access (two simultaneous transitions from same state)
- [ ] T020 [P] Document `MailboxProcessor` backpressure as known limitation (XML doc comment on `MailboxProcessorStore`)

### Implementation Notes
- The `IStateMachineStore` interface is UNCHANGED -- this WP only adds documentation and tests
- `MailboxProcessorStore` already satisfies the contract -- tests validate the existing behavior
- Concurrency test pattern: fire N `SetState` calls via `Task.WhenAll`, then verify final state reflects all N transitions in order
- Backpressure documentation: note `CurrentQueueLength` for monitoring, recommend Kestrel limits as primary mechanism

### Parallel Opportunities
- WP03 can run in parallel with WP04 (SQLite store) -- they are independent concerns
- T019 and T020 can proceed in parallel

### Dependencies
- Depends on WP02 (Types.fs finalized; test helpers may reference updated Guard DU)

### Risks & Mitigations
- Concurrency tests are inherently non-deterministic -- use sufficient iteration count and explicit synchronization primitives (e.g., `ManualResetEventSlim` barrier) to ensure concurrent contention

## Work Package WP04: SQLite Durable Store Implementation (Priority: P2)

**Goal**: Create `Frank.Statecharts.Sqlite` package with `SqliteStateMachineStore` -- an actor-wrapped SQLite implementation of `IStateMachineStore` supporting durable persistence, lazy rehydration, subscriptions, and DI registration.
**Independent Test**: Build the new project. Instantiate the store, set state, get state, verify persistence. Verify schema auto-creation.
**Prompt**: `/tasks/WP04-sqlite-store-implementation.md`
**Estimated Size**: ~500 lines

### Included Subtasks
- [x] T021 Create `Frank.Statecharts.Sqlite` project structure and `.fsproj` (multi-target net8.0;net9.0;net10.0)
- [x] T022 Add project to `Frank.sln`, set up project references and NuGet dependencies (`Microsoft.Data.Sqlite`)
- [x] T023 Implement `SqliteStateMachineStore` actor core: `MailboxProcessor` with `StoreMessage` DU, in-memory cache, subscriber list
- [x] T024 Implement SQLite persistence: schema auto-creation, connection management (single connection, WAL mode, busy_timeout), UPSERT on `SetState`, SELECT on cache miss
- [x] T025 Implement lazy rehydration: `GetState` checks in-memory cache first, loads from SQLite on miss
- [x] T026 Implement Subscribe/observable pattern (in-memory subscriber list, notify after persist, BehaviorSubject semantics)
- [x] T027 Implement `IDisposable` for connection and actor cleanup

### Implementation Notes
- Architecturally identical to `MailboxProcessorStore` but with SQLite persistence inside actor loop
- Single `SqliteConnection` opened once, kept alive for store lifetime (safe because actor is single-threaded)
- JSON serialization via `System.Text.Json` with configurable `JsonSerializerOptions`
- Composite primary key `(instance_id, state_type)` allows multiple state machine types in one database
- SQLite operations are synchronous inside the actor (Microsoft.Data.Sqlite async is synchronous under the hood)
- `state_type` column holds fully qualified .NET type name (e.g., `typeof<'State>.FullName`)

### Parallel Opportunities
- WP04 can run in parallel with WP03 (actor validation) -- they are independent
- T026 (subscriptions) and T027 (IDisposable) can proceed after T023-T025 are complete

### Dependencies
- Depends on WP02 (Types.fs finalized, `IStateMachineStore` interface confirmed unchanged)

### Risks & Mitigations
- F# DU serialization with `System.Text.Json` requires `FSharp.SystemTextJson` -- document this as a soft dependency (users configure their own serializer)
- SQLite file locking by external processes -- `PRAGMA busy_timeout=5000` handles this; `SqliteException` surfaced via `ILogger`
- No test project in this WP -- WP05 handles tests to keep WP04 focused on implementation
