# Work Packages: Cross-Format Statechart Validator

**Inputs**: Design documents from `/kitty-specs/021-cross-format-validator/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/validation-api.fsi
**Dependencies**: Shared Statechart AST (spec 020) must provide `StatechartDocument`, `StateNode`, `TransitionEdge`, etc. in `Frank.Statecharts.Ast` namespace.

**Tests**: Included -- the spec explicitly defines acceptance scenarios and the plan calls for Expecto tests.

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package is independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `/tasks/` generated by `/spec-kitty.tasks`. Treat this file as the high-level checklist; keep deep implementation detail inside the prompt files.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- Include precise file paths or modules.

## Work Package WP03: Self-Consistency Validation Rules (Priority: P1)

**Goal**: Implement universal self-consistency rules that validate structural integrity of any single-format artifact: orphan transition targets, duplicate state identifiers, required AST fields, and isolated state warnings.
**Independent Test**: Each rule produces correct pass/fail/skip results when run against well-formed and malformed test artifacts.
**Prompt**: `/tasks/WP03-self-consistency-rules.md`
**Estimated size**: ~400 lines

### Included Subtasks
- [x] T016 Implement orphan transition target rule (all transition targets reference existing states) in `src/Frank.Statecharts/Validation/Validator.fs`
- [x] T017 Implement duplicate state identifier rule in `src/Frank.Statecharts/Validation/Validator.fs`
- [ ] T018 Implement required AST fields rule (non-empty state identifiers, non-empty transition sources) in `src/Frank.Statecharts/Validation/Validator.fs`
- [ ] T019 Implement isolated state warning rule (states with no incoming/outgoing transitions) in `src/Frank.Statecharts/Validation/Validator.fs`
- [ ] T020 Implement empty statechart warning rule (no states, no transitions) in `src/Frank.Statecharts/Validation/Validator.fs`
- [ ] T021 Create a `SelfConsistencyRules` module exposing a `rules: ValidationRule list` value aggregating all universal rules

### Implementation Notes
- All rules have `RequiredFormats = Set.empty` (universal -- run on every artifact).
- Each rule iterates over every artifact individually (not pairwise).
- Isolated states produce warnings (check status `Pass` with a note), not failures, per spec scenario 1.3.
- Orphan targets, duplicates, and missing required fields produce `Fail` checks with corresponding `ValidationFailure` entries.
- Case-sensitive comparison for all identifiers (decision D-010).

### Parallel Opportunities
- T016-T020 are independent rules that can be developed in parallel.

### Dependencies
- Depends on WP02 (AstHelpers and Validator.validate must exist).

### Risks & Mitigations
- Rule for isolated states must distinguish warnings from failures. Mitigation: Use `Pass` status with a descriptive reason string for warnings, since the spec says "warning (not a failure)".

## Work Package WP04: Cross-Format Validation Rules (Priority: P1)

**Goal**: Implement cross-format pairwise validation rules for state name agreement and event name agreement between format pairs. Define a `CrossFormatRules` module with registerable rules.
**Independent Test**: Cross-format rules detect mismatches between two format artifacts (state names, event names) and pass when artifacts agree.
**Prompt**: `/tasks/WP04-cross-format-rules.md`
**Estimated size**: ~450 lines

### Included Subtasks
- [x] T022 Implement generic state name agreement rule (symmetric check between any two formats) in `src/Frank.Statecharts/Validation/Validator.fs`
- [x] T023 Implement generic event name agreement rule (symmetric check between any two formats) in `src/Frank.Statecharts/Validation/Validator.fs`
- [x] T024 Implement generic transition target agreement rule in `src/Frank.Statecharts/Validation/Validator.fs`
- [x] T025 Implement casing mismatch detection in cross-format comparisons (D-010)
- [x] T026 Create `CrossFormatRules` module exposing `rules: ValidationRule list` for all pairwise format combinations
- [x] T027 Implement rule generation for all 10 pairwise format combinations (5 choose 2)

### Implementation Notes
- Cross-format rules require exactly 2 formats in `RequiredFormats`.
- Generic comparison functions should be parameterized by format pair, then rules are generated for each of the 10 pairs.
- Casing mismatch detection: if `String.Equals(a, b, StringComparison.OrdinalIgnoreCase)` but `a <> b`, include explicit casing note in the failure description.
- Each rule receives the full artifact list and filters by its required format tags.
- Symmetric checks: if A has state "foo" missing from B, AND B has state "bar" missing from A, report both as separate failures.

### Parallel Opportunities
- T022-T024 (state names, event names, transition targets) are independent rule types.
- T025 (casing detection) is a helper used by T022-T024.

### Dependencies
- Depends on WP02 (AstHelpers and Validator.validate must exist).
- Can be developed in parallel with WP03 (both depend on WP02 only).

### Risks & Mitigations
- 10 pairwise combinations generate many rules. Mitigation: Use a generic rule factory that accepts two format tags and produces the rule, avoiding code duplication.
